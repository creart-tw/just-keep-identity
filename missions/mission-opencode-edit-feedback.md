# Mission: Opencode `edit` 工具實戰反饋與技術諮詢 (Consultation)

## 1. 事故案例分享 (The Case Study)

在 `just-keep-identity` 專案中，我們遭遇了一次嚴重的「靜默抹除」事故。Agent 在編輯 `.gitignore` 時，提供的 `oldString` 範圍誤觸了檔案末尾的緩衝區，導致在新增測試路徑的同時，意外刪除了後方受保護的私有目錄清單（`private/`, `data/`）。

**目前的補救措施：**
我們在專案規範中強制要求使用 `expectedOldLineCount` 參數。這確實成功攔截了後續的誤操作，但也揭露了 Agent 在處理「結尾換行符號」與「不可見字元」時的高頻認知偏差。

---

## 2. 技術諮詢問題 (Consultation Questions)

我們希望與 Opencode 團隊探討以下方向，以優化 Agent 的開發效率與系統安全性：

### 2.1 診斷資訊的結構化回傳
目前的 `edit` 工具在校驗失敗時，主要回傳純文字報錯。
*   **詢問**：官方是否有計畫提供結構化的錯誤元數據（JSON）？例如回傳 `actualLineCount` 與匹配區塊的 `lineRange`。
*   **價值**：這能讓 Agent 立即獲得物理事實，精確修正認知偏差，而非依賴盲目的 `read-count` 循環。

### 2.2 關於「不可見字元」與「換行符號」的策略
AI 代理人經常在「`oldString` 是否包含結尾換行符號」上失準。
*   **詢問**：官方對於 LF/CRLF 或末尾換行導致的行數不符，是否有預定的處理邏輯（例如自動診斷並提示「多出一個空白行」）？
*   **邊界**：我們不希望工具過度猜測意圖，但希望能有明確的「格式診斷」反饋。

### 2.3 敏感路徑的保護機制
*   **詢問**：Opencode 是否考慮內建對特定關鍵檔案（如 `.gitignore`, `.env`, `Cargo.toml`）的 Tier 0 (零容忍) 保護策略？
*   **背景**：在這些檔案中，任何 +/- 1 行的誤差都可能代表安全性邊界的崩潰。

---

## 3. 實作建議與期望 (Expectations)

我們期望 `edit` 工具能從「簡單取代」進化為「具備契約校驗能力」的工程工具：
1.  **維持冷酷**：繼續拒絕行數不符的操作。
2.  **熱情回饋**：在報錯時回傳物理座標與格式差異診斷。
